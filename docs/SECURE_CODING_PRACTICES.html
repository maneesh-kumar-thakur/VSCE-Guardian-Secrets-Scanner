🔑 Secret Management

✅ DO

  • Store all secrets in .env files - Never hardcode credentials
  • Use environment variables for: API keys, database passwords, JWT secrets, OAuth tokens, private keys, webhook secrets
  • Rotate secrets regularly: Critical (30 days), High (90 days), After incidents (immediately)
  • Use strong secret values: 32+ characters, mixed case + numbers + symbols, generated by service provider
  • Create separate credentials per environment: Development, Staging, Production
  • Audit secret access - Know who accessed what and when

❌ DON'T

  • ❌ Commit .env files to Git
  • ❌ Hardcode API keys or tokens in source code
  • ❌ Share secrets via Slack, email, or unencrypted channels
  • ❌ Use production secrets in development
  • ❌ Log or console.log secret values
  • ❌ Include secrets in error messages
  • ❌ Reuse the same secret across multiple environments
  • ❌ Store plaintext passwords in comments

💡 Example

✓ GOOD: 
const apiKey = process.env.STRIPE_API_KEY;
const dbUrl = process.env.DATABASE_URL;
✗ BAD: 
const apiKey = "sk_live_1234567890abcdef";
const dbUrl = "postgres://user:password@host:5432/db";

👁️ Code Review Practices

Pre-Commit Review Checklist

  • No hardcoded secrets - Run Guardian to scan
  • No API keys or tokens - Check comments and strings
  • Environment variables used - Sensitive data from .env
  • No debug logging - console.log() with sensitive data removed
  • No commented credentials - Delete all commented passwords/keys
  • Dependencies verified - npm audit passed
  • No new large files - 20MB+ files should be questioned
  • Git history clean - No accidental commits with secrets

PR Review Standards

Security Checklist for Pull Requests:

  1. Secret Exposure Check
    • Run Guardian Scanner
    • Check for hardcoded credentials
    • Verify .env is in .gitignore
  2. Dependency Safety
    • Run npm audit
    • Check for known vulnerabilities
    • Review new dependencies
  3. Data Handling
    • Sensitive data encrypted in transit
    • PII not logged or exposed
    • Proper error handling without details
  4. Access Control
    • API keys scoped to minimum permissions
    • Database credentials read-only if possible
    • Webhook signatures validated

Commit Message Guidelines

✓ GOOD: 
git commit -m "Add AWS configuration

- Configure S3 bucket access
- Use environment variables for credentials
- See docs/CONFIGURATION.md for setup"
✗ BAD: 
git commit -m "Add AWS key: AKIAIOSFODNN7EXAMPLE"

If a Secret Was Accidentally Committed

  1. Immediately rotate the secret! (within 5 minutes)
  2. Clean from git history using git-filter-branch
  3. Force push after review
  4. Notify team and audit access logs

💻 Local Development Security

Safe Development Practices

  • ✅ Use separate user accounts for dev vs. production
  • ✅ Disable unnecessary services (SSH, etc.)
  • ✅ Keep OS and tools updated
  • ✅ Use full-disk encryption on laptop
  • ✅ Lock computer when away from desk
  • ✅ Don't shoulder-surf sensitive information
  • ✅ Use secure WiFi (VPN) for remote work

Pre-commit Hooks

Use Git hooks to prevent secrets from committing:

#!/bin/bash
# .git/hooks/pre-commit

# Run Guardian Scanner
npm run scan

# Check for environment files
if git diff --cached --name-only | grep -qE "\\.env|\\.pem"; then
  echo "❌ ERROR: Attempting to commit sensitive files!"
  exit 1
fi

🚀 CI/CD Security

GitHub Actions Best Practices

✅ DO - Use GitHub Secrets, not hardcoded values 
env:
  API_KEY: ${{ secrets.API_KEY }}
❌ DON'T - Hardcode secrets 
env:
  API_KEY: "sk_live_12345678"

Branch Protection Rules

  • Require status checks to pass (Guardian Scanner, npm test, npm audit)
  • Require pull request reviews before merging
  • Restrict who can push (Admins only)
  • Include administrators in restrictions

📊 Monitoring & Logging

What NOT to Log

❌ Never log these: 
console.log('API Key:', apiKey);
console.error('Database password:', dbPassword);
console.log('User token:', jwtToken);
throw new Error(`Failed with: ${connectionString}`);

Safe Logging

✓ Mask sensitive data: 
const maskSecret = (secret) => {
  if (secret.length <= 8) return '***';
  return secret.substring(0, 4) + 
         '*'.repeat(secret.length - 8) + 
         secret.substring(secret.length - 4);
};

logger.info('API call', { 
  endpoint: 'https://api.stripe.com/v1/charges',
  key: maskSecret(apiKey)  // "sk_t...ample"
});

Onboarding Checklist

  • Provide this security guide
  • Send environment setup instructions privately
  • Add to GitHub team with appropriate permissions
  • Enable 2FA on their account
  • Share secret distribution ceremony
  • Require them to sign security agreement
  • Review with them on day 1

Offboarding Checklist

  • Rotate all secrets they had access to
  • Remove Git push access within 24 hours
  • Revoke API keys they used
  • Remove from sensitive services
  • Audit their recent commits for secrets

🚨 Emergency Response - If a Secret is Exposed

Immediate Actions (within 5 minutes)

  1. 📢 Notify the team in #security channel
  2. 🔄 Rotate the secret immediately in the service provider
  3. 🧹 Clean Git history using git-filter-branch
  4. 🔍 Audit access logs for suspicious activity

Within 24 Hours

  • Document what happened (root cause)
  • Determine who had access
  • Review permissions for principle of least privilege
  • Update security policies if needed
  • Send incident report to leadership
  • Hold post-mortem meeting

❓ FAQ

Q: Can I test with production secrets?

NO Always use test/development keys. If you accidentally use production, rotate immediately.

Q: Is it safe to commit .env.example?

YES Make sure it has NO real values, only templates and comments.

Q: How often should I rotate secrets?

  • Critical (DB passwords): Every 30 days
  • High (API keys): Every 90 days
  • After any incident: Immediately
  • When someone leaves: Immediately

Q: Can I use the same key across environments?

NO Each environment needs its own credentials for access control.
