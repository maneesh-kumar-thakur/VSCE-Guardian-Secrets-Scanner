## Secret Management

### ✅ DO

- **Store all secrets in `.env` files** - Never hardcode credentials
  ```typescript
  // ✅ GOOD
  const apiKey = process.env.STRIPE_API_KEY;
  ```

- **Use environment variables for all sensitive data:**
  - API keys (Stripe, AWS, Google Cloud)
  - Database credentials
  - JWT signing keys
  - OAuth tokens
  - Private keys
  - Webhook secrets

- **Encrypt `.env` files in transit** - Use secure channels to share
  ```bash
  # Use encrypted communication only
  # Never share .env via Slack, email, or pastebins
  ```

- **Rotate secrets regularly**
  - Critical: Every 30 days
  - High: Every 90 days
  - Document rotation in a changelog

- **Use strong secret values**
  - Minimum 32 characters for API keys
  - Mix of uppercase, lowercase, numbers, symbols
  - Generated by your service provider (don't create manually)

- **Create separate credentials per environment**
  ```
  .env.local         → Development (never committed)
  .env.staging       → Staging (encrypted, access-controlled)
  .env.production    → Production (in secure vault only)
  ```

- **Audit secret access** - Know who accessed what secrets and when

### ❌ DON'T

- ❌ Commit `.env` files to Git (even accidentally)
- ❌ Hardcode API keys, passwords, or tokens in source code
- ❌ Share secrets via Slack, email, or unencrypted channels
- ❌ Use production secrets in development
- ❌ Log or console.log secret values
- ❌ Include secrets in error messages or stack traces
- ❌ Reuse the same secret across multiple environments
- ❌ Store plaintext passwords in comments

## Code Review Practices

### Pre-Commit Review Checklist

Every commit should be reviewed before merging. Use this checklist:

- [ ] **No hardcoded secrets** - Run Guardian to scan
- [ ] **No API keys or tokens** - Check comments and strings
- [ ] **Environment variables used** - Sensitive data from `.env`
- [ ] **No debug logging** - console.log() with sensitive data removed
- [ ] **No commented credentials** - Delete all commented passwords/keys
- [ ] **Dependencies verified** - npm audit passed
- [ ] **No new large files** - 20MB+ files should be questioned
- [ ] **Git history clean** - No accidental commits with secrets

### PR Review Standards

When reviewing pull requests:

```markdown
Security Checklist for Code Review:

1. **Secret Exposure**
   - [ ] Run Guardian Scanner on the branch
   - [ ] Check for hardcoded credentials
   - [ ] Verify .env is in .gitignore

2. **Dependency Safety**
   - [ ] Run npm audit
   - [ ] Check for known vulnerabilities
   - [ ] Review new dependencies

3. **Data Handling**
   - [ ] Sensitive data encrypted in transit
   - [ ] PII not logged or exposed
   - [ ] Proper error handling without details

4. **Access Control**
   - [ ] API keys scoped to minimum permissions
   - [ ] Database credentials read-only if possible
   - [ ] Webhook signatures validated
```

### Commit Message Guidelines

**Don't reveal secrets in commit messages:**

```bash
# ❌ BAD
git commit -m "Add AWS key: AKIAIOSFODNN7EXAMPLE"

# ✅ GOOD
git commit -m "Add AWS configuration

- Configure S3 bucket access
- Use environment variables for credentials
- See docs/CONFIGURATION.md for setup"
```

### Branch Protection

**Set these rules on main branch:**

1. **Require pull request reviews** (minimum 1)
2. **Require status checks to pass** before merging
3. **Require branches to be up to date**
4. **Restrict who can push** to main
5. **Run Guardian Scanner** as automated check

### Detecting & Removing Secrets from History

**If a secret was accidentally committed:**

```bash
# 1. Immediately rotate the secret!
# 2. Clean from git history
git-filter-branch --env-filter '
if [[ $GIT_COMMIT_MESSAGE =~ "secret_value" ]]; then
  export GIT_AUTHOR_DATE="$GIT_AUTHOR_DATE"
  export GIT_COMMITTER_DATE="$GIT_COMMITTER_DATE"
fi
' -- --all

# 3. Force push (after review!)
git push origin --force-with-lease --all

# 4. Notify team and audit access logs
```

### Pre-commit Hooks

**Use Git hooks to prevent secrets from committing:**

```bash
# .git/hooks/pre-commit
#!/bin/bash

# Run Guardian Scanner before commit
npm run scan --fix

# Check for environment files
if git diff --cached --name-only | grep -qE "\\.env|\\.pem|secrets\\.json"; then
  echo "❌ ERROR: Attempting to commit sensitive files!"
  echo "Do not commit .env, *.pem, or secrets.json"
  exit 1
fi

# Check for high-entropy strings
if git diff --cached -U0 | grep -qE "^[+].*([A-Za-z0-9+/]{40,})" && ! grep -q "DUMMY\|EXAMPLE\|TEST"; then
  echo "⚠️  WARNING: Possible secret detected (high entropy string)"
  echo "Review and confirm this is not a real credential"
fi

exit 0
```

### Safe Development Practices

- ✅ Use separate user accounts for development vs. production

- ✅ Use full-disk encryption on laptop
- ✅ Lock computer when away from desk
- ✅ Don't shoulder-surf sensitive information
- ✅ Use secure WiFi (VPN) for remote work

## CI/CD Security

### GitHub Actions Security

**Protect secrets in workflows:**

```yaml
name: Security Scan

on: [pull_request, push]

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      # ✅ Use GitHub Secrets, not hardcoded
      - uses: actions/checkout@v3
      
      - name: Run Guardian Scanner
        run: npm run scan
        
      - name: Check for secrets
        run: |
          if npm run scan 2>&1 | grep -i "critical\|high"; then
            echo "❌ Security findings detected!"
            exit 1
          fi
      
      # ✅ Don't log secrets
      - name: Deploy
        run: npm run deploy
        env:
          # Never echo these!
          API_KEY: ${{ secrets.API_KEY }}
          DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
```

### Secrets Management in CI/CD

```yaml
# ✅ DO - Use GitHub Secrets
env:
  API_KEY: ${{ secrets.API_KEY }}

# ❌ DON'T - Hardcode
env:
  API_KEY: "sk_live_12345678"

# ✅ DO - Use production-only workflows
- if: github.ref == 'refs/heads/main'
  name: Deploy to Production
  env:
    PROD_TOKEN: ${{ secrets.PROD_TOKEN }}

# ✅ DO - Mask sensitive output
- name: Login to Registry
  run: echo ${{ secrets.REGISTRY_PASSWORD }} | docker login -u user --password-stdin
```

## Monitoring & Logging

### What NOT to Log

```typescript
// ❌ Never log these
console.log('API Key:', apiKey);
console.error('Database password:', dbPassword);
console.log('User token:', jwtToken);
logger.info('Stripe key:', stripeKey);

// ❌ Never include in error messages
throw new Error(`Failed to connect: ${connectionString}`);

// ❌ Never expose in stack traces
catch (error) {
  res.json({ error: error.message }); // May contain secrets!
}
```

### Safe Logging

```typescript
// ✅ Mask sensitive data in logs
const maskSecret = (secret: string) => {
  if (secret.length <= 8) return '***';
  return secret.substring(0, 4) + 
         '*'.repeat(Math.max(0, secret.length - 8)) + 
         secret.substring(secret.length - 4);
};

logger.info('API call', { 
  endpoint: 'https://api.stripe.com/v1/charges',
  key: maskSecret(apiKey)  // "sk_t...8ample"
});

// ✅ Store logs securely
// - Centralized log service (not in repo)
// - Access control on logs
// - Automatic redaction
// - Retention policies
```

### Audit Trail

**Log important security events:**

```typescript
logger.audit({
  event: 'SECRET_CREATED',
  timestamp: new Date(),
  service: 'stripe',
  creator: 'john.doe@example.com',
  expiresAt: new Date(Date.now() + 90 * 24 * 60 * 60 * 1000)
});

logger.audit({
  event: 'CREDENTIALS_ROTATED',
  timestamp: new Date(),
  service: 'database',
  rotatedBy: 'admin',
  reason: 'quarterly_rotation'
});
```

## Emergency Response

### If a Secret is Exposed

**Immediate actions (within 5 minutes):**

```
1. 🚨 STOP! Don't panic, act quickly.

2. 📢 Notify the team in #security channel:
   "🔴 CRITICAL: AWS_SECRET_KEY exposed in commit abc123"

3. 🔄 ROTATE THE SECRET:
   - Go to AWS Console
   - Deactivate the exposed access key
   - Generate a new one
   - Update .env in all environments

4. 🧹 CLEAN GIT HISTORY:
   - Run git-filter-branch
   - Force push to all branches
   - Notify affected developers

5. 🔍 AUDIT ACCESS:
   - Check AWS CloudTrail for suspicious activity
   - Review API logs for unusual requests
   - Scan for compromises
```

**Within 24 hours:**

- [ ] Document what happened (root cause)
- [ ] Determine who had access
- [ ] Review permissions for principle of least privilege
- [ ] Update security policies if needed
- [ ] Send incident report to leadership
- [ ] Hold post-mortem meeting

## FAQ

### Q: Can I test with production secrets?
**A:** ❌ NO. Always use test/development keys. If you accidentally use production with test code, you need to rotate immediately.

### Q: Is it safe to commit .env.example?
**A:** ✅ YES. Make sure it has NO real values, only templates and comments.

### Q: How often should I rotate secrets?
**A:** 
- **Critical** (like DB passwords): Every 30 days
- **High** (API keys): Every 90 days
- **After any incident**: Immediately
- **When someone leaves**: Immediately

### Q: What if I see a secret in code?
**A:** Report to #security immediately. Don't shame the developer - focus on remediation.

### Q: Can I use the same key across environments?
**A:** ❌ NO. Each environment needs its own credentials so you can revoke access per environment.

### Q: How do I handle secrets in Docker?
**A:** Never bake secrets into images. Use Docker secrets or Kubernetes secrets:
```dockerfile
# ❌ DON'T
ENV API_KEY=sk_live_xxxxx

# ✅ DO
RUN --mount=type=secret,id=api_key \
    cat /run/secrets/api_key > config.env
```
